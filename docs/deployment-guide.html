Environment Configuration (.env)

The root .env file provides secrets to Docker Compose. Never commit this file to source control.

.env
# SQL Server SA password — must match across all services
SA_PASSWORD=RetailERP@2024!

# JWT signing secret — minimum 32 characters, keep secret
JWT_SECRET=RetailERP-SuperSecret-JWT-Key-2024-Must-Be-At-Least-32-Characters

# Grafana admin password
GRAFANA_PASSWORD=admin

These values are injected into each container via the x-common-env YAML anchor in docker-compose.yml:

docker-compose.yml (excerpt)
x-common-env: &common-env
  ASPNETCORE_ENVIRONMENT: "Docker"
  ASPNETCORE_URLS:         "http://+:8080"
  ConnectionStrings__DefaultConnection: "Server=sqlserver;Database=RetailERP;User Id=sa;Password=RetailERP@2024!;TrustServerCertificate=true"
  Jwt__Secret:    "RetailERP-SuperSecret-JWT-Key-2024..."
  Jwt__Issuer:    "RetailERP.Auth"
  Jwt__Audience:  "RetailERP.Client"

Each microservice uses the same parameterized Dockerfile.api with build arguments SERVICE_NAME and SERVICE_DIR.

  --build-arg SERVICE_NAME=Inventory \
  --build-arg SERVICE_DIR=inventory \
  -t retailerp-inventory:latest \

docker build \

  -t retailerp-frontend:latest \
  .

# Build the Gateway image
docker build \

  .

Production Hardening for Docker

  • Use Docker secrets or an external secrets manager (HashiCorp Vault, AWS Secrets Manager) instead of plain .env files.
  • Replace the shared SA password with a dedicated application database user with minimal privileges.
  • Add a reverse proxy (Nginx, Traefik, or Caddy) in front of the frontend and gateway containers to handle SSL termination.
  • Set ASPNETCORE_ENVIRONMENT=Production in all microservice containers (disables Swagger UI and developer error pages).
  • Enable Redis password authentication and set --requirepass in the Redis command.
  • Configure log drivers (json-file with rotation, or forward to a centralized log system) to prevent disk fill.
  • Use container resource limits (mem_limit, cpus) to prevent any one service from exhausting host resources.
  • Generate a strong JWT secret: openssl rand -base64 48.
docker-compose.override.yml — production limits example
services:
  inventory-api:
    deploy:
      resources:
        limits:
          cpus: '1.0'
          memory: 512M
        reservations:
          memory: 256M
    logging:
      driver: "json-file"
      options:
        max-size: "50m"
        max-file: "5"

Step 1 — SQL Server Setup

  1. Install SQL Server 2022 (Standard or Developer edition).
  2. Enable SQL Server Authentication mode (Mixed Mode) and set the sa password to RetailERP@2024! (or your chosen password).
  3. Create the database and run all initialization scripts from the database/ folder in order:
PowerShell — run SQL scripts
$server   = "localhost"
$user     = "sa"
$password = "RetailERP@2024!"
$dbname   = "RetailERP"
$sqlcmd   = "sqlcmd"

# Create database
& $sqlcmd -S $server -U $user -P $password `
  -Q "IF NOT EXISTS (SELECT * FROM sys.databases WHERE name='$dbname') CREATE DATABASE $dbname"

# Run all scripts in order
$scriptFolders = @('schemas', 'tables', 'indexes', 'stored-procedures', 'seed-data')
foreach ($folder in $scriptFolders) {
  Get-ChildItem ".\database\$folder\*.sql" | ForEach-Object {
    Write-Host "Running $_"
    & $sqlcmd -S $server -U $user -P $password -d $dbname -i $_.FullName
  }
}

Step 3 — Publish Microservices

Publish each microservice as a self-contained deployment from the build machine (requires .NET SDK 8.0).

PowerShell — publish all services
$services = @(
  @{ name='Auth';       dir='auth' },
  @{ name='Product';    dir='product' },
  @{ name='Inventory';  dir='inventory' },
  @{ name='Order';      dir='order' },
  @{ name='Production'; dir='production' },
  @{ name='Billing';    dir='billing' },
  @{ name='Reporting';  dir='reporting' },
  @{ name='Gateway';    dir='gateway' }
)

foreach ($svc in $services) {
  $proj = if ($svc.name -eq 'Gateway') {
    "src\services\gateway\RetailERP.Gateway.csproj"
  } else {
    "src\services\$($svc.dir)\RetailERP.$($svc.name).API\RetailERP.$($svc.name).API.csproj"
  }
  dotnet publish $proj `
    -c Release `
    -r win-x64 `
    --self-contained false `
    -o "C:\RetailERP\services\$($svc.name)"
  Write-Host "Published $($svc.name)"
}

Step 4 — IIS Application Pools & Sites

Create one Application Pool per service. All pools should use No Managed Code (ASP.NET Core runs out-of-process).

PowerShell — create app pools and sites
Import-Module WebAdministration

$services = @(
  @{ name='Auth';       port=5001 },
  @{ name='Product';    port=5002 },
  @{ name='Inventory';  port=5003 },
  @{ name='Order';      port=5004 },
  @{ name='Production'; port=5005 },
  @{ name='Billing';    port=5006 },
  @{ name='Reporting';  port=5007 },
  @{ name='Gateway';    port=5000 }
)

foreach ($svc in $services) {
  $poolName = "RetailERP-$($svc.name)"
  $siteName = "RetailERP-$($svc.name)"
  $physPath = "C:\RetailERP\services\$($svc.name)"

  # Create App Pool (No Managed Code for ASP.NET Core)
  New-WebAppPool -Name $poolName
  Set-ItemProperty "IIS:\AppPools\$poolName" `
    -Name managedRuntimeVersion -Value ""

  # Create IIS Site
  New-Website -Name $siteName `
    -ApplicationPool $poolName `
    -PhysicalPath $physPath `
    -Port $svc.port `
    -Force
}

Step 5 — web.config for Each Microservice

Each published microservice needs a web.config in its publish directory. The ASP.NET Core module (ANCM) is installed by the .NET Hosting Bundle.

C:\RetailERP\services\Inventory\web.config
<?xml version="1.0" encoding="utf-8"?>
<configuration>
  <system.webServer>
    <handlers>
      <add name="aspNetCore"
           path="*"
           verb="*"
           modules="AspNetCoreModuleV2"
           resourceType="Unspecified" />
    </handlers>
    <aspNetCore
        processPath="dotnet"
        arguments=".\RetailERP.Inventory.API.dll"
        stdoutLogEnabled="true"
        stdoutLogFile=".\logs\stdout"
        hostingModel="inprocess">
      <environmentVariables>
        <environmentVariable name="ASPNETCORE_ENVIRONMENT" value="Production" />
        <environmentVariable name="ConnectionStrings__DefaultConnection"
                             value="Server=localhost;Database=RetailERP;User Id=retailerp_app;Password=AppPass@2024!;TrustServerCertificate=true" />
        <environmentVariable name="Jwt__Secret"
                             value="YOUR-PRODUCTION-JWT-SECRET-HERE" />
        <environmentVariable name="Jwt__Issuer"    value="RetailERP.Auth" />
        <environmentVariable name="Jwt__Audience"  value="RetailERP.Client" />
      </environmentVariables>
    </aspNetCore>
  </system.webServer>
</configuration>

<?xml version="1.0" encoding="utf-8"?>
<configuration>
  <system.webServer>
    <handlers>
      <add name="iisnode"
           path="server.js"
           verb="*"
           modules="iisnode" />
    </handlers>
    <rewrite>
      <rules>
        <rule name="Node" stopProcessing="true">
          <match url=".*" />
          <conditions logicalGrouping="MatchAll" trackAllCaptures="false">
            <add input="{REQUEST_FILENAME}" matchType="IsFile" negate="true" />
          </conditions>
          <action type="Rewrite" url="server.js" />
        </rule>
      </rules>
    </rewrite>
    <iisnode
      nodeProcessCommandLine="node"
      loggingEnabled="true"
      logDirectory="logs\"
      debuggingEnabled="false" />
  </system.webServer>
  <appSettings>
    <add key="PORT"                    value="3000" />
    <add key="NEXT_PUBLIC_API_URL"     value="http://localhost:5000" />
  </appSettings>
</configuration>

Step 8 — IIS Reverse Proxy (ARR)

Use Application Request Routing (ARR) to expose the frontend on port 80/443 and proxy API requests to the gateway.

C:\inetpub\wwwroot (or dedicated site)\web.config — ARR reverse proxy
<?xml version="1.0" encoding="utf-8"?>
<configuration>
  <system.webServer>
    <rewrite>
      <rules>
        <!-- Proxy /api/* to the YARP Gateway on port 5000 -->
        <rule name="API Gateway Proxy" stopProcessing="true">
          <match url="^api/(.*)" />
          <action type="Rewrite"
                  url="http://localhost:5000/api/{R:1}" />
        </rule>
        <!-- Proxy all other traffic to Next.js frontend on port 3000 -->
        <rule name="Frontend Proxy" stopProcessing="true">
          <match url="(.*)" />
          <action type="Rewrite"
                  url="http://localhost:3000/{R:1}" />
        </rule>
      </rules>
    </rewrite>
    <security>
      <requestFiltering>
        <requestLimits maxAllowedContentLength="52428800" /> <!-- 50 MB -->
      </requestFiltering>
    </security>
  </system.webServer>
</configuration>
⚠️
Enable ARR Proxy
After installing ARR, you must enable the proxy in IIS Manager: open Application Request Routing Cache at the server level → click Server Proxy Settings → check Enable proxy → click Apply.

Step 9 — SSL / TLS Configuration

Bind an SSL certificate to the main IIS site (port 443) and redirect HTTP to HTTPS.

PowerShell — bind SSL certificate
# Import your certificate (.pfx) into the Windows Certificate Store
$cert = Import-PfxCertificate `
  -FilePath "C:\certs\retailerp.pfx" `
  -CertStoreLocation "Cert:\LocalMachine\My" `
  -Password (ConvertTo-SecureString "certPassword" -AsPlainText -Force)

# Bind HTTPS to port 443
New-WebBinding -Name "RetailERP-Main" `
  -Protocol "https" -Port 443 -HostHeader "yourdomain.com"

$sslBinding = Get-Item "IIS:\SslBindings\0.0.0.0!443" -ErrorAction SilentlyContinue
if (-not $sslBinding) {
  $cert | New-Item "IIS:\SslBindings\0.0.0.0!443"
}
web.config — HTTP to HTTPS redirect (add to main site)
<rewrite>
  <rules>
    <rule name="HTTP to HTTPS" stopProcessing="true">
      <match url="(.*)" />
      <conditions>
        <add input="{HTTPS}" pattern="off" ignoreCase="true" />
      </conditions>
      <action type="Redirect"
              url="https://{HTTP_HOST}/{R:1}"
              redirectType="Permanent" />
    </rule>
  </rules>
</rewrite>

Jwt__ExpiryInHours1–24Shorter = more secure; users re-authenticate more often
🔐
JWT Secret Must Be Identical Across All Services
Every microservice validates JWT tokens using the same secret. If any service has a different Jwt__Secret, it will reject tokens issued by the Auth service, causing 401 Unauthorized errors. Use environment variables or a shared secret store to ensure all services share the exact same value.

Troubleshooting
Common issues and their resolutions for Docker and IIS deployments

Docker Common Issues

SymptomCauseResolution
Container exits immediately Missing DLL, wrong SERVICE_NAME build arg, or startup crash Run docker compose logs <service> for the error. Verify build args match the actual project name.
SQL Server stays "starting" Insufficient memory (SQL Server needs 2 GB minimum) Increase Docker Desktop memory limit to 8 GB or more.
401 on all API calls JWT secret mismatch between auth and other services, or CORS misconfiguration Verify Jwt__Secret is identical in all service containers. Check Cors__Origins in gateway config.
Frontend shows "Network Error" NEXT_PUBLIC_API_URL points to wrong host Rebuild frontend image with correct env var. In Docker, the browser calls localhost:5000 directly; ensure gateway port is exposed.
db-init container keeps restarting SQL Server not healthy yet when db-init starts db-init depends on service_healthy; if SQL Server health check fails, increase retries or start_period in docker-compose.yml.
Port already in use Another process holds the port Run netstat -ano | findstr :5001 (Windows) or ss -tlnp | grep 5001 (Linux) to find and stop the conflicting process.
Images fail to build NuGet restore failures or no internet access Run docker compose build --no-cache --progress=plain for full output. Check proxy settings if on a corporate network.

IIS Common Issues

SymptomCauseResolution
HTTP 500.19 — Config error ANCM not installed or web.config syntax error Reinstall the .NET 8 Hosting Bundle. Validate web.config XML syntax.
HTTP 502.5 — Process failure The dotnet process crashed on startup Enable stdoutLogEnabled="true" in web.config and check the logs\stdout_*.log file for the exception.
HTTP 500.30 — Start failure Missing connection string, wrong DLL name, or missing .NET runtime Verify the arguments value in web.config matches the actual DLL filename. Verify .NET 8 runtime is installed.
Application Pool stops Rapid failure restarts (5 failures in 5 minutes triggers stop) Open IIS Manager → Application Pools → set Rapid Fail Protection to disabled during debugging. Fix the underlying startup error first.
Frontend shows blank page iisnode not installed or Node.js not found Verify iisnode.dll is registered in IIS. Ensure Node.js path is in the system PATH variable. Restart IIS (iisreset).
ARR not proxying Proxy not enabled in ARR server settings IIS Manager → server root → Application Request Routing Cache → Server Proxy Settings → check Enable proxy.
CORS errors in browser Frontend domain not in Cors__Origins Add the production domain to Cors__Origins in gateway and auth service environment variables.

Database Connectivity Issues

SymptomCauseResolution
Login failed for user 'sa' SQL Server Authentication not enabled, or wrong password In SQL Server Management Studio: Server Properties → Security → select "SQL Server and Windows Authentication mode". Restart SQL Server service.
Cannot connect to server SQL Server TCP/IP protocol disabled SQL Server Configuration Manager → SQL Server Network Configuration → Protocols for MSSQLSERVER → enable TCP/IP. Restart SQL Server service.
Database tables not found db-init scripts did not run or partially failed Run the SQL scripts manually from the database/ folder in order: schemas → tables → indexes → stored-procedures → seed-data.
Connection timeout in Docker Service started before SQL Server was ready The health check ensures SQL Server is ready before services start. If issues persist, increase start_period for sqlserver in docker-compose.yml.

Pre-Deployment Checklist

  • Changed SQL Server SA password from default RetailERP@2024!
  • Generated a new JWT secret (minimum 48 characters, base64 encoded)
  • Updated Grafana admin password from admin
  • Changed the application admin password (Admin@123) after first login
  • Set ASPNETCORE_ENVIRONMENT=Production on all services (disables Swagger)
  • Verified all 7 microservices respond to /health endpoint
  • Verified database initialization completed (all schemas and seed data)
  • SSL certificate installed and HTTPS redirect configured
  • CORS origins updated to the production domain
  • Log rotation configured (Docker json-file driver or Windows Event Log)
  • Firewall rules restrict direct access to microservice ports (5001–5007)
  • Only ports 80 and 443 exposed publicly; all others internal only
  • Backup schedule configured for SQL Server database
  • Monitoring alerts configured in Grafana for service downtime
